# 証跡と完了ゲート

この文書は、UI/UX変更を完了扱いするために必要な証跡と判定条件を定義します。

## 1. 原則

UI/UX変更では、主張だけでは不十分です。次のような証跡が必要です。

- 実際に変更された画面・コンポーネントの説明
- スクリーンショット、録画、トレース、または取得不能理由
- UI変更があるPRでは、該当画面・状態ごとの変更前スクリーンショットと変更後スクリーンショットをPR本文に添付する
- state matrix
- 初見シミュレーション
- ユーザー価値評価
- アクセシビリティ確認
- 視覚階層確認
- コピー確認
- 熟練者効率確認
- 満足感・信頼感確認
- 反証レビュー
- 実行したテスト・検証
- 実行していないテスト・検証
- 残リスク

## 2. 必須成果物

### 2.1 ユーザー価値評価

次を明示します。

- 対象ユーザー
- ユーザー目的
- 支援するタスク
- UIがなければ困る点
- UIが助ける理解・判断・行動
- 価値が弱い場合の代替案

### 2.2 state matrix

次の状態を、該当しない理由がない限り確認します。

- 通常
- 読み込み中
- 空
- 検索結果なし
- 部分データ
- エラー
- 入力エラー
- 無効
- 権限不足
- オフラインまたは利用不可
- 狭幅
- 文字拡大
- 長文・大量データ

### 2.3 初見シミュレーション

初見ユーザーとして、画面目的、現在地、最初の行動、予測される結果、失敗時の回復を確認します。

### 2.4 アクセシビリティ確認

自動検査だけでなく、少なくとも主要タスクのキーボード操作、フォーカス、ラベル、名前、状態通知を確認します。

### 2.5 熟練者効率確認

主要反復タスクについて、手数、再入力、再選択、毎回表示される説明、ショートカット、一括操作、復帰導線を確認します。

### 2.6 満足感・信頼感確認

成功、待機、失敗、危険操作、データ操作、権限、個人情報、送信、公開、削除のUIが、ユーザーに安心できるフィードバックを返しているか確認します。

### 2.7 反証レビュー

実装を落とすつもりで、P0が残っていないか確認します。

## 3. 完了ゲート

UI/UX変更は、次をすべて満たした時だけ完了扱いできます。

- P0が残っていない。
- ユーザー価値を説明できる。
- 初見理解が成立している。
- state matrixが作成されている。
- アクセシビリティ最低確認が実施されている。
- 視覚階層とコピーが確認されている。
- 熟練者効率が確認されている。
- 満足感・信頼感が確認されている。
- 反証レビューが実施されている。
- 証跡が提出されている。
- UI変更があるPRでは、該当する前後スクリーンショットがPR本文で確認できる。
- 実行していない検証が明示されている。
- PR がある場合は、CI 完了後の Codex 自動コードレビュー確認結果と、未解決 review thread の有無が明示されている。

## 4. 証跡を出せない場合

スクリーンショットや自動検査を実行できない場合、次を報告します。

- 実行できなかった検証
- 実行できなかった理由
- 代替として確認した内容
- 残るリスク
- 後続で必要な確認

「実行できなかったが問題ない」は禁止です。

UI変更PRの前後スクリーンショットを取得できない場合も同じ扱いです。取得不能理由と代替証跡をPR本文に書くだけでは完了扱いにせず、残るリスクと次に必要な確認を明示してください。

## 5. 推奨される検証

リポジトリに環境がある場合は、次を使います。

- lint
- typecheck
- unit test
- integration test
- end-to-end test
- Storybook
- Playwrightなどのブラウザテスト
- axe-coreなどのアクセシビリティ検査
- スクリーンショット差分
- キーボード操作確認
- レスポンシブ確認
- 文字拡大確認

環境がない場合は、ないことを明示します。
