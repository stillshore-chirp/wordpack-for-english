# UI/UXレビュー報告: Lexicon一覧の適用中検索・絞り込み条件 2026-07-24

## 1. 概要

- 変更した画面・コンポーネント: Lexicon の保存済み WordPack 一覧見出し、件数表示、適用中条件、上部検索との同期
- 判定: Issue #164 の範囲は Pass。Lexicon 一覧全体は Issue #115 の P1 が残るため継続
- P0件数: 0
- P1件数: 全体で1（Issue #115）
- P2件数: 0

## 2. ユーザー価値

- 対象ユーザー: 検索・公開状態・生成状態を組み合わせて保存済み WordPack を繰り返し探すユーザー
- 利用文脈: 条件適用後に一覧を走査する、下へスクロールする、同じセッションで一覧へ戻る、前回条件が復元される場面
- ユーザー目的: 何を対象にした一覧か、なぜ件数が減ったか、どの条件を外せばよいかを判断する
- 支援するタスク: 条件確認、検索方式確認、個別解除、全解除、件数範囲の判断
- このUIが助ける理解・判断・行動: 上部検索欄と複数の押下状態を見比べず、一覧見出し直下の1領域で条件と解除方法を確認できる
- このUIがなければ困る点: セッション復帰時に前回条件を覚えていないと、データ欠落や取得不具合と誤認し、各コントロールを順番に調べる必要がある
- 削るべき情報・操作: 該当なし状態カード内に重複していた条件一覧と解除ボタンを除き、常設の条件領域を唯一の解除起点にした
- 検証仮説・成功指標: 検索・公開・生成の3条件と検索方式が一覧領域で読め、各条件を1操作、全条件を1操作で解除できる。復元時も検索欄と条件表示が一致する

## 3. 初見理解

- 何の画面か分かるか: 「Lexicon」「保存済みWordPack」を維持する
- 今どこか分かるか: 左ナビゲーションの Lexicon 選択状態と、一覧見出し直下の「適用中の条件」で対象範囲を示す
- 何ができるか分かるか: 条件ラベルの隣に「解除」、領域右上に「すべて解除」を可視表示する
- 最初の有意味な行動: 表示結果が期待と違う場合、条件を読み、不要な条件の「解除」を押す
- 操作結果を予測できるか: `検索: alpha（部分一致）`、`公開状態: 非公開`、`生成状態: 生成済み` と、解除対象を文言で示す
- 失敗時に戻れるか: 個別解除後は残る条件見出し、全解除後は保存済み一覧見出しへフォーカスを移し、現在の文脈を失わない

## 5. アクセシビリティ確認

- キーボード: 各「解除」と「すべて解除」は native button で Tab、Enter、Space 操作ができる
- フォーカス: 個別解除後は残る条件見出し、最後または全解除後は保存済み一覧見出しへ移す。見出しには可視アウトラインを付ける
- 名前・ラベル: 個別操作は「解除」を可視表示し、`検索: alpha（部分一致）を解除` のように条件全文を accessible name と title にする
- 見出し・構造: 条件領域を labelled section、条件集合を named `ul` とし、条件を `li` で表す
- コントラスト: シアン境界と暗色背景に加え、見出し、本文、ラベルで意味を示す。色だけに依存しない
- ターゲットサイズ: 個別解除と全解除は最小44px
- エラー・ステータス: 条件数とこのページの一致件数は polite live region。エラー自体は #545 の alert を維持する
- 自動検査: desktop複数条件と390px復元条件で Playwright + axe 違反0件
- 手動確認: 固定モックの1280×831変更前後画像で、情報階層、折り返し、条件とカードの対応を確認した

## 6. 視覚階層

- 主操作: 「すべて解除」を領域右上に置き、個別解除は各条件ラベルと一体化する
- 情報優先度: 一覧見出しと件数、適用中条件、絞り込みコントロール、並び順、項目の順
- グルーピング: 条件の見出し・一致件数、全解除、個別条件を1枚の境界内へまとめる
- 余白・密度: 3条件を2行以内に折り返し、カードの走査領域を過度に圧迫しない
- 読みやすさ: 長い検索語は `overflow-wrap:anywhere` で条件境界内に折り返す
- 狭幅・文字拡大: 760px以下では見出しと全解除を縦積み、各条件を全幅にする

## 7. コピー

- 用語: 「適用中の条件」「全体」「このページ」「条件一致」で対象範囲と状態を明示する
- ボタン・リンク: 全解除は「すべて解除」。個別解除のaccessible nameには条件種別・値・検索方式を含める
- エラー文: 今回は変更せず、#545 の原因・影響・回復文言を維持する
- 空状態: 該当なしカードは、上の条件領域から個別または全解除できることだけを説明し、条件一覧を重複表示しない
- disabled: 条件解除はローカル状態変更のため待機不要
- トーン: データ欠落と誤認させず、現在の条件を事実として短く示す

## 8. 熟練者効率

- 主要反復タスク: 条件を組み合わせて一覧を狭め、対象を開く
- 手数: 解除前のコントロール探索をなくし、個別・全解除を1操作にする
- 再入力・再選択: セッション復元条件を維持し、検索欄にも適用済み検索語を同期する
- 近道: 既存の検索 Enter、`⌘/Ctrl+K`、フィルターボタンを維持する
- 初心者向け説明の影響: 条件がある場合だけ領域を出し、通常一覧の垂直空間を増やさない
- 判定: Pass

## 9. 満足感・信頼感

- 待機中: 復元条件を隠さず、取得中も対象範囲を確認できる
- 成功時: 条件解除と同時に一致件数と項目が更新される
- 失敗時: 条件と取得失敗を別領域にし、原因を混同しない
- 危険操作: 条件解除はデータを削除せず、保存項目を再表示する可逆操作
- データ・権限・個人情報: 証跡は固定モックのみで、ユーザー提供画像や実データを公開ファイルへ転用しない
- トーン: 「欠落」ではなく「条件適用中」「このページで一致」と説明する
- 判定: Pass

## 10. 反証レビュー

- 実装を落とす観点で見つけた問題: 常設条件領域を追加した直後、#545 の状態カードにも解除操作が残り、同じ操作が重複して720px表示で後者が初期表示外へ押し出された。状態カード側の重複条件・解除操作を除き、常設領域を唯一の回復起点にして #545 E2E を載せ替えた
- P0候補: 条件解除ボタンが記号だけに見える案を棄却し、可視テキスト「解除」、条件ラベルと同一境界、44px、全文accessible name、title、フォーカス表示を追加した。該当なし時も解除操作が初期表示内にあることを回帰検証した
- 証跡不足: 実ユーザーによる条件探索時間、スクリーンリーダー実機、非常に長い日本語検索語の実機入力は未実施
- 残リスク: 現在の条件と内訳は取得済みページを対象にする。201件以上を含む全件検索・集計は Issue #115 でAPI/UIを一体修正する
