## 2. ユーザー価値

- 対象ユーザー: 201件以上の保存済み WordPack から、見出し語、公開状態、例文生成状態を組み合わせて探すユーザー
- 利用文脈: 先頭ページに対象がない、後続ページから条件を変更する、該当なしの理由を判断する場面
- ユーザー目的: 保存済み全体から対象を漏れなく見つけ、表示件数の範囲を誤認しない
- 支援するタスク: 前方・後方・部分一致検索、公開・生成状態の組合せ、並び替え、次ページ移動、条件解除
- このUIが助ける理解・判断・行動: 「全体」「このページ」「条件一致（全ページ）」を分離し、絞り込みボタンの数字が切替後の全ページ件数であることを明記する
- このUIがなければ困る点: 201件目だけに一致する WordPack が存在しても「該当なし」と表示され、データ欠落と誤認する
- 削るべき情報・操作: 現在ページだけを対象にした重複クライアント集計を正本にせず、API が返す全ページ件数を表示の基準にする
- 検証仮説・成功指標: 201件目だけの一致が先頭ページの操作から表示される。条件変更後は offset 0 に戻り、全体件数と条件一致件数が常に別ラベルで一致する

## 10. 反証レビュー

- 実装を落とす観点で見つけた問題: サーバー0件レスポンスでは `items=[]` のため、従来判定のままだと「初めて使う空状態」になった。`total=0` と `filtered_total=0` を分離し、保存済み全体がある0件は該当なし状態へ固定した
- ページ境界: 条件変更後に保存済み offset を使うと、条件一致1件でも offset 200 の空ページになり得る。検索・絞り込み・並び順変更は offset 0 で再取得し、削除等で現在ページが範囲外になった場合は最後の有効ページへ補正する
- 件数の反証: 公開中1件を選んだとき非公開200件も表示されるため、現在結果の内訳と誤読し得た。「数字は切替後の件数」を可視表示し、API文書にも facet の条件保持規則を記載した
- 遷移中の反証: 条件を切り替えてAPI応答を待つ間、以前の `filtered_total` が新条件の確定件数に見えた。query key を照合し、件数とfacetを「確認中」へ切り替え、0件状態とページングも新応答まで確定表示しない
- 遷移失敗の反証: 新条件の取得に失敗したあと、前回の行を新条件でクライアント再絞り込みすると、説明は「前回一覧を表示」なのにカードが消え、件数も「確認中」のままになった。成功済み query key と現在 key が違う間は前回の行順とカードを保持し、失敗後は条件を「未反映」、件数を「未取得」として、再試行までページングを無効にした
- P0候補: 201件目を取得しても全体件数だけ201・一致件数0のままなら誤誘導になる。API/UI/E2Eで `filtered_total=1` とカード1件の同時更新を固定した
- 証跡不足: 実ユーザーテスト、スクリーンリーダー実機、数千件規模の本番データ性能は未実施
- 残リスク: 後方・部分一致の正しさを優先し、現在は認可範囲の軽量 WordPack 一覧文書を読み込んでサーバー内で条件適用する。個人辞書規模では許容するが、規模拡大時は読み取り数・p95を監視し、検索専用索引を検討する
