## 2. ユーザー価値

- 対象ユーザー: 保存済み WordPack を確認・管理するログイン済みユーザー、および公開 WordPack を閲覧するゲスト
- 利用文脈: 初回利用、検索・絞り込み後、通信失敗、既存一覧の再読み込み中
- ユーザー目的: 一覧に項目がない理由を判断し、データを失ったと誤認せず、最短の回復操作へ進む
- 支援するタスク: 新規作成、検索解除、絞り込み解除、初回読み込み再試行、更新再試行
- このUIが助ける理解・判断・行動: 「未作成」「条件不一致」「取得失敗」「既存データを保持した更新失敗」を見出し、説明、条件、操作で区別できる
- このUIがなければ困る点: 取得失敗時にも「保存済みのWordPackがありません」と表示され、保存データが消えたように見える。検索結果0件では一覧領域が空白になり、原因と解除方法が分からない
- 削るべき情報・操作: データがない状態で無効な表示切替、絞り込み、並び順を隠し、回復に必要な操作だけを残す
- 検証仮説・成功指標: 主要状態が固有の見出しを持ち、初回空は新規作成、該当なしは条件解除、失敗は再試行へ1操作で進める。更新中・更新失敗でも前回一覧が消えない

## 3. 初見理解

- 何の画面か分かるか: 「Lexicon」「保存済みWordPack」の見出しを維持している
- 今どこか分かるか: 左ナビゲーションの Lexicon 選択状態を維持している
- 何ができるか分かるか: 状態ごとに「新しいWordPackを作成」「検索を解除」「絞り込みを解除」「もう一度読み込む」「更新を再試行」を1つの主操作として示す
- 最初の有意味な行動: 初回空では新規作成、該当なしでは条件解除、失敗では再試行
- 操作結果を予測できるか: ボタン文言を状態に対応させ、該当なしでは解除対象の条件も表示する
- 失敗時に戻れるか: 初回失敗は再取得できる。更新失敗は前回一覧を残したまま再試行できる

## 5. アクセシビリティ確認

- キーボード: すべての回復操作は native button。空状態の作成操作で見出し語入力へフォーカスを移す
- フォーカス: 検索解除後も上部検索欄の値を同期して空にする。再試行後は通常一覧へ遷移する
- 名前・ラベル: 状態見出しを `aria-labelledby` で section に関連付け、適用中条件を「現在適用中の条件」というリスト名で公開した
- 見出し・構造: 状態ごとに固有の h3 を持ち、単なる空白や色だけで原因を伝えない
- コントラスト: 通常・該当なし・失敗を左境界色と記号に加えて見出し・本文でも区別する
- ターゲットサイズ: 回復ボタンは最小 44px 高。狭幅では全幅にする
- エラー・ステータス: 空・該当なしは polite status、失敗は assertive alert。読み込みは既存 status を使う
- 自動検査: 主要4フローで Playwright + axe 違反 0 件
- 手動確認: 安全な固定モックの変更後5画像を目視し、内容、順序、折り返し、前回一覧保持を確認した

## 6. 視覚階層

- 主操作: 各状態に1つの強い回復操作だけを配置
- 情報優先度: 記号と見出し、影響説明、詳細または条件、回復操作の順
- グルーピング: 状態メッセージ、条件、操作を別領域に分ける
- 余白・密度: データがないときは一覧操作群を隠し、状態カードの可読幅を確保する
- 読みやすさ: 詳細は本文より弱く、ただし失敗理由を隠さない。適用中条件は短いチップで表示する
- 狭幅・文字拡大: 1320px 以下で条件・操作の左インデントを外し、回復操作を全幅にする

## 7. コピー

- 用語: 初回空は「まだありません」、条件不一致は「一致する…がありません」、取得失敗は「読み込めませんでした」と原因を分ける
- ボタン・リンク: 操作結果が分かる動詞を含め、「OK」「戻る」のような曖昧な文言を使わない
- エラー文: 保存データが削除されたわけではないこと、再読み込み失敗時は表示が最新でない可能性を明示する
- 空状態: ログイン時は見出し語登録の価値と作成導線、ゲスト時は公開項目の待機状態を示す
- disabled: 取得中は更新を無効にし、読み込み状態を同時に表示する
- トーン: ユーザーを責めず、不要なデータ損失不安を抑え、回復方法を同じ領域に置く

## 8. 熟練者効率

- 主要反復タスク: 一覧を更新し、検索・絞り込みし、対象を開く
- 手数: 通常状態の操作手数は増やさない。該当なしから1操作で一覧へ戻る
- 再入力・再選択: 更新中・更新失敗でも既存一覧、表示方法、条件を保持する
- 近道: 上部検索の既存 Enter 適用と `⌘/Ctrl+K` フォーカスを維持
- 初心者向け説明の影響: 状態説明は異常・空状態にだけ表示し、通常一覧の反復走査を妨げない

- 待機中: 初回は何を確認中か示し、更新時は一覧を消さず「表示したまま」と明示する
- 成功時: 状態カードを取り除いて通常一覧と正しい件数へ戻す
- 失敗時: データ消失と通信失敗を区別し、保持データが古い可能性も隠さない
- 危険操作: 今回は変更しない
- データ・権限・個人情報: 証跡は固定モックのみを使い、ユーザー提供画像や実データを公開ファイルへ転用しない
- トーン: 不安を煽らず、原因・影響・回復を同じ場所で説明する
- 判定: Pass

## 10. 反証レビュー

- 実装を落とす観点で見つけた問題: リクエスト回数だけでテスト応答を切り替えると React の重複初期化で失敗状態が即座に上書きされる。テストを画面状態ベースのモックへ変更し、実装側も request id で古い応答・abort の loading/error 更新を抑止した。さらに同一1280×831の比較で、検索0件時にも未絞り込みの「最近開いたWordPack」が残り、状態の意味を曖昧にして解除ボタンを初期表示外へ押し出す問題を検出したため、検索・絞り込み中は最近項目を隠し、解除操作が初期表示内に収まることをE2Eで固定した
- P0候補: 初回失敗時に従来の空状態が併存しないこと、更新中・更新失敗で前回一覧が消えないことを unit と E2E の双方で再確認した。残る P0 はなし
- 証跡不足: 実ユーザーによる回復操作時間、スクリーンリーダー実機読み上げ、Windows 実機は未実施
- 残リスク: 一覧 API は最大200件単位であり、ページをまたぐ検索・絞り込みと件数整合は Issue #115、通常時の常設条件サマリーと個別解除は Issue #164 に残る
